# 性能リファクタリング計画（既存機能を壊さない方針）

## 実施済み（要約）

- **Phase 0:** `docs/RELEASE_CHECK.md` に手動チェックリストを追加。
- **Phase 1:** `router.tsx` で各ページを `React.lazy` + `Suspense` に変更。ルートごとに chunk 分割。
- **Phase 4:** `DashboardPage` の unmount 時に Undo 用タイマーを clear する cleanup を追加。
- **Phase 2:** `tasksRepo` に `listTasksByDateRange(startDate, endDate)` を追加。`DayTasksDrawer` でその日のタスクのみ取得するように変更（`tasks` prop を削除）。`DashboardPage` / `PlanningPage` の呼び出しを更新。
- **Phase 3:** `react-virtuoso` を導入。`WorkspacePage` のグループボード・プロジェクトボードの列内タスクリストを `Virtuoso` で仮想スクロール化。

**補足:** `npm run build` でアセット（JS/CSS）は正常に出力される。Service Worker 書き込みで `workbox-build` の "Unfinished hook action (terser) renderChunk" が出る環境があるが、本リファクタとは無関係の既知事象。必要なら `vite-plugin-pwa` の issue やビルド環境の調整を検討。

---

## 目的

性能レビューで指摘された問題を、**既存の挙動・機能を変えずに**段階的に解消する。  
各フェーズは「動かなくならないこと」を前提にスコープを切り、検証可能な単位で進める。

---

## 前提・制約

- **既存機能は維持する**  
  画面の見た目・操作フロー・データの保存/復元・バックアップ/インポートは現状どおり動くこと。
- **段階的**  
  一度に複数箇所を触らず、フェーズごとに「変更範囲」と「受け入れ条件」を決めてから実施する。
- **ロールバック可能**  
  各フェーズは独立して revert できる単位にする（大きな一枚岩の変更にしない）。
- **CI は常に通す**  
  lint / typecheck / test / build は各フェーズ完了時点で成功すること。

---

## 問題の整理（性能レビューに基づく）

| 領域 | 問題 | 影響 |
|------|------|------|
| 初回ロード | ルート単位のコード分割なし。全ページが初回バンドルに含まれる | 初回 JS サイズ・解析コストが増える |
| ランタイム | リストを全件 DOM に展開。仮想スクロール未使用 | タスク数増でスクロール・操作が重くなる |
| ランタイム | フィルタ・ソートを全件メモリで実施 | データ増でメインスレッドが重くなる |
| IndexedDB | `listTasks()` が全件 `toArray()`。ページング・DB 側絞り込みなし | 読み込み時間・メモリ使用量がデータ量に比例 |
| メモリ | Undo 用タイマーの unmount 時クリーンアップなし | 遷移時にタイマーが残る可能性（軽微） |

キャッシュ・イベントリスナー・Toast タイマーは現状で問題なしのため、本計画では触れない。

---

## フェーズ概要

| フェーズ | 内容 | リスク | 既存機能への影響 |
|----------|------|--------|-------------------|
| 0 | 準備（検証方法・必要なら手動チェックリスト） | 低 | なし |
| 1 | ルート単位のコード分割（lazy load） | 低 | なし（表示・遷移は同一） |
| 2 | IndexedDB：タスク取得の段階的改善（新 API 追加→呼び出し移行） | 中 | なし（既存 API は残す） |
| 3 | 仮想スクロールの導入（1 画面ずつ） | 中 | なし（見た目・操作は同等に保つ） |
| 4 | その他（Undo タイマー cleanup 等） | 低 | なし |

---

## Phase 0: 準備

### 目標

リファクタ前後の「動いていること」を確認する手段を決め、問題が起きたときにすぐ気づけるようにする。

### スコープ

- **含める**
  - 主要画面・主要操作の**手動チェックリスト**を 1 つ用意する（例: ダッシュボード表示、タスク追加/編集/削除/完了、ワークスペース表示・フィルタ、Planning、設定のエクスポート/インポート、マスタ編集など）。
  - 必要なら `docs/` に「リリース前チェック」のような短文ドキュメントを追加する。
  - 既存の CI（lint / typecheck / test / build）がそのまま使えることを確認する。
- **含めない**
  - 自動 E2E の新規導入は必須にしない（既存テストがあれば維持）。

### 受け入れ条件

- [ ] チェックリスト（または短文ドキュメント）が存在し、誰でも「主要機能が動いているか」を確認できる。
- [ ] CI が現状どおり成功する。

### ロールバック

ドキュメント追加のみのため、削除すればよい。

---

## Phase 1: ルート単位のコード分割（Lazy Load）

### 目標

初回ロード時に不要なページの JS を読まないようにし、初回バンドルを小さくする。**画面の表示・遷移・データの扱いは変えない。**

### スコープ

- **含める**
  - `router.tsx` で各ルートコンポーネントを `React.lazy()` で import する。
  - ルート要素を `<Suspense fallback={...}>` で包む（fallback はローディング表示や最小限のプレースホルダでよい）。
  - 既存の `DashboardPage` / `PlanningPage` / `WorkspacePage` / `SettingsPage` の**中身は変更しない**（import の仕方だけ変える）。
- **含めない**
  - コンポーネント内部の分割や、新しいデータ取得ロジックの追加。
  - デザイン変更。

### 受け入れ条件

- [ ] 各ルート（/dashboard, /planning, /workspace, /settings）に遷移したとき、これまでどおり表示・操作できる。
- [ ] ビルド成果物に、ルートごとの chunk（例: Dashboard-xxx.js 等）が分かれて出力されている。
- [ ] CI が通る。

### 注意点・リスク

- 遅延ロードにより、初回でそのルートの chunk を取るわずかな待ちが発生する。fallback で体感を悪化させないこと。

### ロールバック

`router.tsx` の `React.lazy` と `Suspense` をやめ、従来の直接 import に戻す。

---

## Phase 2: IndexedDB タスク取得の段階的改善

### 目標

「全件 toArray()」に依存しない取得経路を用意し、将来的にページングや DB 側フィルタ（日付範囲・ステータス等）を使えるようにする。**既存の `listTasks()` は残し、挙動を変えない。**

### スコープ

- **含める**
  - **新規 API の追加のみ**（例: `listTasksByDateRange(start, end)` や `listTasksPaginated(offset, limit)` など）。既存の `listTasks()` はそのまま全件 `filter().toArray()` で維持する。
  - 新 API を「使うかどうか」はフェーズ内で決めてよい。まずは 1 画面（例: カレンダーの日付範囲だけ）で新 API を試し、問題がなければ他画面へ広げる。
  - 既存の `listTasks()` を呼んでいる箇所は、**移行する場合のみ**新 API に差し替える。移行時も「取得結果の内容・順序・表示」が既存と同等になるようにする。
- **含めない**
  - 既存の `listTasks()` のシグネチャ変更や、全呼び出しの一括置き換え（一括は Phase 2 後半か別フェーズで検討）。
  - スキーマ変更（必要なら別計画でバージョンアップ）。

### 受け入れ条件

- [ ] `listTasks()` の既存の呼び出しは、変更しない限り従来どおり全件取得で動く。
- [ ] 新 API を導入した画面では、表示・フィルタ・ソートがこれまでと同等に動く（データの欠損・重複・順序の不整合がない）。
- [ ] CI が通る。

### 進め方の提案

1. **Step 2a**  
   `tasksRepo` に `listTasksByDateRange(startDate, endDate)`（または必要な範囲クエリ）を追加。IndexedDB の `where('dueDate')` 等を利用し、範囲内だけ返す。  
   呼び出し側はまずは 1 箇所（例: カレンダー週/日表示で使うデータ）だけ新 API に切り替え、手動で表示・操作を確認。
2. **Step 2b**  
   問題がなければ、他画面で「全件いらない」箇所を同様に新 API へ移行。  
   全画面を移行しきらない場合は、残りは従来どおり `listTasks()` のままでもよい（段階的移行）。

### ロールバック

新 API を呼んでいる箇所を `listTasks()` に戻し、新 API 関数は未使用のまま残すか削除する。

## Phase 3: 仮想スクロールの導入

### 目標

タスク一覧など、リストが長くなりうる箇所で DOM ノード数を抑え、スクロール・操作が重くならないようにする。**表示内容・操作（クリック、編集、削除、ドラッグ等）は既存と同等に保つ。**

### スコープ

- **含める**
  - 仮想スクロール用ライブラリの選定と導入（例: `react-window` または `react-virtuoso`。プロジェクト方針に合わせて 1 つに統一）。
  - **1 画面ずつ**仮想リストに差し替える。候補順:
    1. タスク一覧が主な画面（例: Dashboard 内のタスクリスト、または Tasks 系画面があればその一覧）
    2. Workspace の列内タスクリスト
    3. その他、`.map(task => ...)` で長大リストになっている箇所
  - 各画面で「リストが短いとき」も正しく表示されるようにする（高さ 0 や空リストの扱い）。
- **含めない**
  - ドロップダウン（グループ・プロジェクト・Bucket 選択）などの短いリストの仮想化（優先度低）。
  - 見た目やインタラクションの仕様変更（「同じ操作で同じ結果」を維持）。

### 受け入れ条件

- [ ] 仮想リストを入れた画面で、従来どおりスクロール・クリック・編集・削除・完了トグル等ができる。
- [ ] タスクが 0 件・1 件・多数件のいずれでも、表示と操作がおかしくならない。
- [ ] CI が通る。

### 注意点・リスク

- ドラッグ＆ドロップやキーボードフォーカスを使っている場合、仮想リストと相性を確認する。必要なら該当画面は後回しにする。
- 行の高さが可変の場合は、ライブラリの「可変高さ」対応（例: react-virtuoso の VariableSizeList 的なもの）を検討する。

### ロールバック

該当画面だけ仮想リストをやめ、従来の `.map()` による通常のリストに戻す。

---

## Phase 4: その他（メモリ・軽微な修正）

### 目標

性能レビューで指摘された軽微な点を解消する。既存機能には影響させない。

### スコープ

- **含める**
  - **Dashboard の Undo 用タイマー**  
    `undoTimerRef` にセットした `setTimeout` を、コンポーネントの unmount 時に `useEffect` の cleanup で `clearTimeout` する。
  - 上記以外で、レビューで「軽微」とされたメモリまわりの修正（必要なら同様に cleanup の追加のみ）。
- **含めない**
  - 既存の仕様・UI の変更。
  - 大規模なリファクタ。

### 受け入れ条件

- [ ] Undo 表示・取り消し操作は従来どおり動く。
- [ ] 該当コンポーネントをマウント→アンマウントしたときに、タイマーが残って動作しないこと（必要なら手動で確認）。
- [ ] CI が通る。

### ロールバック

追加した cleanup を削除すればよい。

---

## 実施順序と依存関係

```
Phase 0（準備）
    ↓
Phase 1（コード分割）  ← 他フェーズに依存しない
    ↓
Phase 4（Undo タイマー）  ← 短時間で完了可能
    ↓
Phase 2（IndexedDB）  ← Phase 3 より先にやると、仮想リストで「表示するデータ」を減らしやすい
    ↓
Phase 3（仮想スクロール）  ← データ取得をある程度整理した後にやると安全
```

- Phase 0 のあと、**Phase 1 と Phase 4 は並行してもよい**（別ファイルを触るため）。
- Phase 2 と Phase 3 は、どちらを先にするかは「日付範囲で取る」など DB 側を先に整えるか、「まず表示を軽くする」かを優先するかで決めてよい。上記は「DB を先に整えてから仮想リスト」の例。

## 既存機能の確認観点（全フェーズ共通）

各フェーズの完了時、少なくとも以下が崩れていないことを確認する。

- **ダッシュボード**  
  表示、カレンダー切り替え（日/週/月）、タスクの追加・編集・完了・削除、Undo、日付クリックで DayTasksDrawer、進捗表示。
- **ワークスペース**  
  表示モード切替、グループ/プロジェクト列、タスクの表示・移動・編集・削除、フィルタ。
- **Planning**  
  表示、日/週/月/年切替、フィルタ、タスク操作。
- **設定**  
  エクスポート（JSON ダウンロード）、インポート（既存データ上書き）、警告表示。
- **マスタ（グループ・プロジェクト・Bucket）**  
  一覧表示、追加・編集・削除、プロジェクトのグループ紐づけ。
- **繰り返しテンプレート**  
  一覧、作成・編集・有効/無効、テンプレートに紐づくタスクの生成。

Phase 0 で上記をチェックリスト化し、Phase 1〜4 の前後で同じリストで確認するとよい。

## ドキュメントの更新

- 各フェーズで「データの取り方」や「ルート構成」を変えた場合、`docs/` や `CLAUDE.local.md` で説明している部分があれば、あとから 1 行〜数行で追記する。
- 新規 API（`listTasksByDateRange` 等）を追加したら、`tasksRepo` または `docs/` に用途・引数の意味を短く書いておく。

## まとめ

| フェーズ | やること | やらないこと |
|----------|----------|----------------|
| 0 | 検証方法・チェックリストの整備 | コード変更・仕様変更 |
| 1 | ルートの lazy load と Suspense | コンポーネント中身の変更 |
| 2 | タスク取得の新 API 追加と、必要に応じた呼び出し移行 | 既存 `listTasks()` の破壊的変更 |
| 3 | 長大リストへの仮想スクロール（1 画面ずつ） | 見た目・操作の仕様変更 |
| 4 | Undo タイマー等の cleanup 追加 | 仕様・UI の変更 |

この順序で進めれば、既存機能を動かさずに性能面のリファクタを段階的に実施できる。
